# レビュー観点カタログ（review perspectives）

dev-loop 内蔵のレビューエンジンが使う**観点の正典**。各観点は `dev-reviewer` サブエージェント 1 体に割り当てられ、
担当観点**のみ**でステップの差分をレビューする。外部プラグインに一切依存しない自己完結のカタログ。

- `dev-state.json` の `review.perspectives[]` は **このファイルのキー** を列挙する（空配列 = 全観点）。
- 各 reviewer は指摘を `[must] / [ask] / [suggestion] / [nits]` のラベル付きで返し、各指摘に
  **安定キー** `key = sha1(perspective + "|" + file + "|" + ルール名 + "|" + 要約)` の短縮（先頭 8 桁）を付ける。
  キーは round 間でオシレーション（直したはずの指摘の再燃）を検知するために使う。
- 「担当外の観点に踏み込まない」「指摘には必ず `file:line` と理由を添える」「問題が無ければ指摘を空にしサマリのみ返す」。

## ラベルの意味（全観点共通）

| ラベル | 意味 | 既定の収束影響 |
|---|---|---|
| `[must]` | 必須修正。これがある限り収束しない（`blockOn` 既定）。 | ブロックする |
| `[ask]` | 確認が必要。仕様不明・判断待ち。`blockOn: "ask"` のときのみブロック。 | 設定次第 |
| `[suggestion]` | 改善提案。スコープ次第でフォローアップ化。 | ブロックしない |
| `[nits]` | 軽微（好み）。修正を強制しない。 | ブロックしない |

`[must]` は「危ない実装を止める」ものに限定する。好みは `[nits]`。「好み」と「良し悪し」を明確に区別する。

---

## 観点一覧

> 既定セット（`plan-init` が scaffold する `perspectives[]`）: `principles` / `architecture` / `code-quality` /
> `correctness` / `testing` / `security` / `performance` / `ui-ux` / `documentation` / `ai-generated`。
> `pr-basics` / `process` / `quality-model` は PR/SDLC メタ寄りで、1 ステップ差分レビューでは既定オフ（必要なら追加）。

### `principles` — レビューの大原則（常に適用 / マクロ視点）
個別バグではなく、変更が**コードベースの健全性**に寄与するかを俯瞰評価する。
- このステップの差分はコードベース全体の健康状態を改善しているか（完璧でなくても改善方向か）。
- **変更容易性**（将来の変更・拡張が最小差分で済む構造か）が最重要。
- 設計意図が差分から読み取れるか（コードはコミュニケーション手段。属人化を生まないか）。
- 「動くか」より「壊れにくいか」。危ない実装を止めることが最大価値。
- 「好み」と「良し悪し」を区別し、好みは `[nits]` に留める。理由を必ず添える。

### `pr-basics` — 変更の基本事項（既定オフ）
- ステップのゴール/成功条件に対しスコープが適切か（過大・過小・無関係変更の混入が無いか）。
- コミットメッセージ・規約への準拠。環境差異（環境変数・設定）に起因する見落としが無いか。
- 依存追加・バージョン指定が妥当か。

### `architecture` — 設計・アーキテクチャ（新クラス/構造変更/DB/依存/外部連携があるとき適用）
- 変更容易性・疎結合高凝集・単一責務（SRP）・適切な抽象化レベル・YAGNI・DRY。
- 既存パターン/慣習との一貫性。継承より委譲。関心の分離（SoC）。実装ではなく抽象/インターフェースに依存。
- 抽象化レベルの統一（SLAP: 1 ルーチン内で高レベルと低レベル詳細を混在させない）。OCP。
- データモデル/スキーマ: 正規化・制約（NOT NULL・FK・unique）・インデックス・命名の一貫性・DB レベルの整合性保証。
- API: リソース指向・インターフェースの一貫性・パラメータ必要十分・内部 ID 等の露出無し・標準形式。
- 依存: 追加ライブラリの妥当性（保守状況/ライセンス/サイズ/セキュリティ）・設定の外部化・車輪の再発明回避。
- **設計レビュー必須ケースの警告 `[must]`**: 後戻り困難な設計 / 外部プロダクト連携 / DB 追加変更 / 新規モデルクラス
  → 人間（シニア）による設計レビューが要る旨をエスカレーション（`requiresHuman` 相当）。

### `code-quality` — コード品質・可読性・保守性・堅牢性
- 命名（意図が伝わる・驚き最小・副作用の明示）。マジックナンバー/値の意味付け。
- 可読性（早期リターン・ネスト削減・適切な関数分割）。重複の共通化。
- 対称性/同型性（似た処理は似た形で書く）。防御的プログラミング（想定外入力・境界の扱い）。
- データ型・不変性（適切な型選択・破壊的変更の局所化・予期せぬ共有状態が無いか）。
- エラーハンドリングの一貫性（握り潰し・無言の失敗が無いか）。

### `correctness` — 機能・仕様・正しさ（ロジック/分岐/状態遷移/新機能があるとき適用）
- ステップの**成功条件**を実際に満たしているか。要件・仕様への準拠。
- 既存機能へのデグレが無いか。エッジケース・異常系の処理。
- 条件分岐が全パターンで正しいか。状態遷移が正しく不正遷移が起きないか。
- diff だけでなくファイル全体・システム全体への影響を確認したか（コンテキスト確認）。
- 想定外入力（型違い・桁あふれ・負値・null・空）でも仕様上意図した振る舞いになるか。

### `testing` — テスト（プロダクションコード変更があるとき適用）
- 変更/新規ロジックに見合うテストがあるか。リスクの高い経路ほど厚く。
- テスト品質（F.I.R.S.T: Fast / Independent / Repeatable / Self-validating / Timely）。テスト間が独立か。
- 境界値・特異値（0・空・最大/最小・境界±1）・状態網羅・異常系（外部依存の失敗）を突いているか。
- アサーションが意味のある検証になっているか（実装の写経・常に通るテストになっていないか）。
- デザイン/実装詳細に過度依存した脆いテストになっていないか。

### `security` — セキュリティ（入力処理/認証認可/外部 I/O/依存変更があるとき適用）
- インジェクション（SQL/コマンド/テンプレート）・XSS・CSRF・パストラバーサル等の脆弱性。
- 認証・認可（権限チェックの欠落・IDOR・過剰な権限）。
- 機密情報（資格情報・トークン・鍵）のハードコードやログ出力・差分混入が無いか。
- 入力バリデーション/サニタイズ。安全でないデシリアライズ・SSRF。
- 依存のバージョン/既知脆弱性・セキュリティアップデート。
- **要注意**: 資格情報の設定や鍵のローテーション等インフラ側の対応は `requiresHuman` 寄りとして明示する。

### `performance` — パフォーマンス（ループ/クエリ/大量データ/ホットパスがあるとき適用）
- N+1 クエリ・不要な反復・計算量（アルゴリズムの O 記法）。
- DB 側で処理すべきものをアプリ側でやっていないか（集計・絞り込み）。インデックス活用。
- メモリ使用（不要な全件ロード・巨大コピー）。同期処理にすべき/非同期にすべきの選択。
- フェールトレランス（外部依存の遅延・タイムアウト・リトライ・サーキットブレーカ）。

### `ui-ux` — UI/UX・フロントエンド（フロント変更があるとき適用）
- HTML 標準要素/セマンティクスの活用（独自実装で再発明していないか）。アクセシビリティ（ラベル・フォーカス・コントラスト・キーボード操作）。
- レスポンシブ・状態（loading/empty/error）の網羅。不要な再レンダリング。
- デザイン仕様（指定があれば）への忠実性。デザイン非依存なテスト構造。

### `documentation` — ドキュメント（公開 API/設定/手順/挙動変更があるとき適用）
- README・API ドキュメント・CHANGELOG・セットアップ手順が変更に追従しているか。
- 関連ドキュメントの同時更新漏れ。
- 「言い訳コメント」（直すべきものをコメントで誤魔化していないか）。コメントは why を語っているか。

### `ai-generated` — AI 生成コード固有（AI 生成の兆候があるとき適用 / 迷えば一部該当）
- **理解の負債**: 「このコードを説明できるか」。誰も説明できないコードは `[must]`。
- **意図の負債**: 要件と実装解釈のズレが無いか（成功条件と突合）。
- 冗長性・不要な抽象化・ボイラープレートの混入。既存パターン/規約との一貫性（驚き最小）。
- 車輪の再発明（標準ライブラリ/既存ヘルパーで足りるか）。YAGNI 違反（早すぎる汎用化）。
- 「テストが通る == 正しい」ではない。AI 生成こそ厳密に検証する。

### `process` — レビュープロセス・コミュニケーション（既定オフ）
- スコープが大きすぎないか（レビュー可能な粒度か）。「あとで修正」を作っていないか。
- 不明点を放置せず `[ask]` で確認しているか。コンテキスト（背景・前提）が共有されているか。

### `quality-model` — 品質保証の多層モデル（既定オフ / SDLC 視点）
- 当該変更の品質保証がどの層（静的解析 / 単体 / 結合 / E2E / 運用監視）で担保されるべきか、層の責任分担が妥当か。
- 継続的改善の観点で、今回見送る品質投資があるなら明示されているか。
